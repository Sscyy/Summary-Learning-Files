# 梯度累积 / 梯度缓存 / 梯度检查点

> 三种显存优化技术的完整知识点总结

---

## 一、梯度累积（Gradient Accumulation）

### 解决的问题

```
想要 batch=64 的训练效果
但显存只够跑 batch=16

解决方案：跑 4 次 batch=16，梯度加起来，再更新一次参数
```

### 核心代码

```python
optimizer.zero_grad()

for i, batch in enumerate(dataloader):
    loss = model(batch)
    loss = loss / accumulation_steps    # 除以累积步数，保证梯度 scale 正确
    loss.backward()                     # 梯度累积到参数上，不清零

    if (i + 1) % accumulation_steps == 0:
        optimizer.step()                # 累积够了才更新
        optimizer.zero_grad()           # 更新完才清零
```

### 为什么要除以 accumulation_steps？

```
真实 batch=64 的 loss：
  L = (1/64) * Σ loss_i     （cross_entropy 内部对 batch 求平均）

梯度累积，4次 batch=16：
  每次 loss_i = (1/16) * Σ loss_j
  4次累积后梯度 = ∂L0/∂θ + ∂L1/∂θ + ∂L2/∂θ + ∂L3/∂θ
               = 4 × (1/16) * Σ
               = 4 × 单次 loss 的梯度

  不除以4 → 梯度是真实 batch=64 的 4 倍 → 等效学习率放大 4 倍 → 训练不稳定

除以 accumulation_steps=4 之后：
  每次 loss = (1/16) * Σ / 4 = (1/64) * Σ
  4次累积 = (1/64) * Σ(全部64个) ✓  和真实 batch=64 完全等价
```

### 显存情况

```
真实 batch=64：前向保存 64 个样本的激活值 → 大

梯度累积 batch=16 × 4：
  每次前向只保存 16 个样本的激活值
  每次 backward 后激活值释放
  梯度累积在参数的 .grad 上（和参数等大，始终存在）

显存峰值 = 1个mini-batch的激活值 + 模型参数 + 参数梯度
```

## 二、梯度缓存（GradCache）

### 解决的问题

对比学习的铁律：**负样本越多，模型越好**。

```
想要大 batch（更多负样本）→ 需要更多显存 → 显存不够

根本原因：大 batch 的激活值太大
  batch=32, seq_len=512, hidden_size=1536, 32层, float16:
  32 × 512 × 1536 × 32 × 2 = 3.2GB  仅激活值
```

### 数学基础

链式法则拆分：

```
∂L/∂θ = ∂L/∂e × ∂e/∂θ

∂L/∂e：只需要 embedding，不需要激活值  → 先算
∂e/∂θ：需要激活值                    → 后算，每次只算一个 mini-batch
```

### 三阶段详解

#### 阶段一：无梯度前向，收集所有 embedding

```python
def _forward_no_grad(self, inputs):
    with torch.no_grad():               # 不建立计算图，不保存激活值
        outputs = self.model(**inputs)
        pooled = mean_pooling(outputs)  # [B, D]

    pooled = pooled.detach().requires_grad_(True)
    # detach()：断开与模型参数的计算图连接
    # requires_grad_(True)：让 pooled 成为叶子节点，可以接收梯度
    # 效果：pooled 可以接收来自损失的梯度，但梯度不会流向模型参数
    return pooled
```

`detach().requires_grad_(True)` 的计算图变化：

```
正常：  θ → 激活值 → pooled → loss
                        ↑ 梯度可以流到这里，也继续流到激活值和 θ

detach 后：  θ → 激活值 → [断开] pooled → loss
                                   ↑ 梯度流到这里就停了
```

对 gc_num_steps 个 mini-batch 都做一遍：

```python
for batch in batches:   # 比如 gc_num_steps=4
    image_emb_list.append(_forward_no_grad(batch['image_inputs']))
    text_emb_list.append(_forward_no_grad(batch['text_inputs']))
# 此时：保存了 4×2 个 detached embedding，无任何激活值，显存极小
```

#### 阶段二：大 batch 算损失，得到 embedding 的梯度

```python
image_embeddings = torch.cat(image_emb_list, dim=0)  # [B*gc_n, D]
text_embeddings  = torch.cat(text_emb_list,  dim=0)  # [B*gc_n, D]

loss = contrastive_loss(image_embeddings, text_embeddings)  # 等价于 batch=64
loss.backward()
# image_embeddings.grad = ∂loss/∂image_embeddings   shape [B*gc_n, D]
# 模型参数的 grad 仍然是 None（embedding 是叶子节点，梯度到此为止）

# split 梯度（不是 split loss！）
image_grad_list = image_embeddings.grad.split(B, dim=0)
# image_grad_list[0] = ∂loss/∂e0   shape [B, D]
# image_grad_list[1] = ∂loss/∂e1   shape [B, D]
# ...
```

**为什么 split 梯度而不是 split loss？**

```
loss 是标量，没法 split。
每个 mini-batch 只能用自己对应的那部分梯度：

  pooled_0 = model(x0)，只有 ∂e0/∂θ 存在
  ∂e1/∂θ = 0，∂e2/∂θ = 0，∂e3/∂θ = 0

  用别人的梯度：∂loss/∂e1 × ∂e1/∂θ = ∂loss/∂e1 × 0 = 0
  没有任何意义，只能用自己的那份。
```

#### 阶段三：带梯度的重新前向，梯度传回模型参数

```python
def _forward_with_grad(self, inputs, cached_grad):
    outputs = self.model(**inputs)      # 正常前向，保存激活值
    pooled = mean_pooling(outputs)      # [B, D]
    pooled.backward(cached_grad)        # 用缓存梯度作为种子
    # 链式法则：∂loss/∂θ += ∂loss/∂pooled × ∂pooled/∂θ
    #                         ↑                  ↑
    #                    cached_grad       计算图自动算

for img_inputs, img_grad in zip(image_inputs_list, image_grad_list):
    _forward_with_grad(img_inputs, img_grad)
    # 每次只保存 1 个 mini-batch 的激活值
    # backward 后激活值释放，梯度累积到 θ.grad 上
    # 注意：这里没有 zero_grad()，梯度是累积的！
```

### 显存峰值对比

```
设：seq_len=512, hidden_size=1536, 32层, float16

普通 batch=64 的激活值：
  64 × 512 × 1536 × 32 × 2 = 6.4GB

GradCache（gc_num_steps=4, mini_batch=16）：
  阶段一：无激活值
  阶段二：只有 embedding [64, D]，可忽略
  阶段三：每次只有 16 个样本的激活值
    16 × 512 × 1536 × 32 × 2 = 1.6GB

显存峰值从 6.4GB 降到 1.6GB，降低 75%
```

### 梯度等价性

**数学上与真实 batch=64 完全等价：**

```
真实 batch=64：
  e = model(x_all)
  ∂loss/∂θ = ∂loss/∂e × ∂e/∂θ

GradCache：
  ∂loss/∂θ = Σ ∂loss/∂ei × ∂ei/∂θ
            = ∂loss/∂e0×∂e0/∂θ + ∂loss/∂e1×∂e1/∂θ + ...
            = 链式法则完整展开 ✓
```

**唯一例外：Dropout**

```
阶段一（no_grad）：dropout mask A → e0_stage1
阶段三（有梯度）：dropout mask B ≠ A → e0_stage3

e0_stage1 ≠ e0_stage3，链式法则断掉，梯度不严格等价。
但 dropout 率通常 0.1，工程上影响可忽略。
严格等价需要两次前向使用相同的随机种子。
```

### 与梯度累积的本质区别

```
梯度累积：
  每次前向建立完整计算图，保存激活值
  loss 是每个 mini-batch 自己算的
  负样本池 = mini-batch 大小，没有扩大

GradCache：
  第一遍前向不保存激活值
  loss 是用拼接后的大 embedding 算的
  负样本池 = 大 batch 大小，扩大了

对比学习用 GradCache，分类/生成任务用梯度累积。
```

## 三、梯度检查点（Gradient Checkpointing）

### 先理解正常反向传播的显存问题

```
前向传播时，每一层的输出（激活值）必须全部保存
因为反向传播时要用激活值来计算梯度

32层 Transformer，batch=16:
  layer1_output  [16, seq_len, D]  ← 保存
  layer2_output  [16, seq_len, D]  ← 保存
  ...
  layer32_output [16, seq_len, D]  ← 保存
  32个激活值同时存在显存里 → 显存大
```

### 核心思路：用计算换显存

```
不保存中间激活值，只保存几个"检查点"
反向传播需要某层激活值时，从最近的检查点重新前向计算
```

### 具体机制

```
正常前向（保存所有激活值）：
  x → L1 → a1 → L2 → a2 → L3 → a3 → L4 → a4 → loss
            ↑         ↑         ↑         ↑
          保存       保存       保存       保存

梯度检查点（只保存检查点，丢弃中间激活值）：
  x → L1 → a1 → L2 → [丢弃] → L3 → a3 → L4 → [丢弃] → loss
            ↑                        ↑
          检查点                   检查点

反向传播需要 a2 时：
  从检查点 a1 重新前向：a1 → L2 → a2（重新计算）
  用 a2 算完梯度后丢弃

反向传播需要 a4 时：
  从检查点 a3 重新前向：a3 → L4 → a4（重新计算）
```

### PyTorch 使用方式

```python
from torch.utils.checkpoint import checkpoint

# 普通前向
output = layer(input)

# 梯度检查点前向（不保存激活值，反向时重新计算）
output = checkpoint(layer, input)
```

HuggingFace 模型开启方式：

```python
model.gradient_checkpointing_enable()
# 内部对每个 Transformer block 使用 checkpoint 包装
```

### 显存对比

```
正常训练（32层）：
  同时保存 32 个激活值

梯度检查点（每层都是检查点）：
  前向时只保存当前层的输入
  反向时重新计算当前层，用完即丢
  同时只存 1 个激活值

显存节省：约 O(层数) 倍
代价：多跑一次前向传播，训练时间增加约 30-40%
```

### 检查点粒度的 tradeoff

```
粒度越细（每层都是检查点）→ 显存越小，重计算越多，时间增加越多
粒度越粗（只有输入是检查点）→ 显存越大，重计算越少

极端情况（只有输入是检查点）：
  相当于重新跑完整前向
  显存 ≈ 只存输入
  时间增加 ~100%（跑了两次完整前向）

常用策略：每个 Transformer block 作为一个检查点
  显存降低 ~layer数 倍，时间增加 ~33%
```

## 四、三者对比总结

| 技术 | 核心思路 | 显存节省来源 | 代价 | 负样本池 | 适用场景 |
|------|---------|------------|------|---------|---------|
| 梯度累积 | 多次 backward 累积梯度 | 每次只跑小 batch | 多次 backward，时间略增 | 不扩大 | 分类、生成任务 |
| GradCache | 分离 ∂L/∂e 和 ∂e/∂θ 的计算 | 第一遍前向无激活值 | 多一次前向 | **扩大** | 对比学习 |
| 梯度检查点 | 丢弃激活值，反向时重新计算 | 不保存中间激活值 | 多一次前向，时间+30% | 无关 | 模型很深/序列很长 |

### 三者可以叠加使用

```
梯度检查点（减少单次前向的激活值显存）
  + 梯度累积（进一步模拟更大 batch）
  + GradCache（对比学习专用，扩大负样本池）

= 在有限显存下最大化训练效果
```

### 选择指南

```
任务是对比学习？
  → 用 GradCache（同时解决显存和负样本问题）

任务是分类/生成，想模拟大 batch？
  → 用梯度累积

模型很深，单次前向显存就爆了？
  → 用梯度检查点

以上都有？
  → 三者叠加
```
