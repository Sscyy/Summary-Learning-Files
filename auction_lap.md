# Auction Lap 竞拍算法详解

## 1. 问题定义

给定 N 个数据点（job）和 K 个聚类中心（worker），要求将每个数据点分配给某个聚类中心，且每个中心**恰好**分配 `N/K` 个数据点（N 必须是 K 的整数倍）。

目标是使所有分配的**总距离最小**，同时满足均衡约束。

这本质上是一个**带均衡约束的线性分配问题（Balanced Linear Assignment Problem）**。

---

## 2. 核心思想

把分配问题类比成一场**拍卖**：

| 拍卖概念 | 对应含义 |
|---------|---------|
| 买家（worker） | 聚类中心，想抢到离自己最近的数据点 |
| 商品（job） | 数据点，被聚类中心竞争 |
| 商品价格（cost） | 某数据点被竞争的激烈程度，竞争越激烈价格越高 |
| 出价（bid） | 某中心为抢某数据点愿意支付的增量 |

每轮迭代，所有未满额的买家同时出价，价高者得。被抢走的买家下一轮继续出价抢其他商品，直到所有商品都被分配。

---

## 3. 算法流程（规范化描述）

**输入**：
- 距离矩阵 `D ∈ R^{N×K}`，`D[j][w]` 表示数据点 j 到中心 w 的距离
- `jobs_per_worker = N / K`，每个中心需要分配的数据点数

**初始化**：
```
score[j][w] = -D[j][w]        # 距离越近，得分越高
cost[j] = 0                    # 每个数据点的当前价格，初始为0
assignment[j] = -1             # 所有数据点未分配
count[w] = 0                   # 每个中心当前持有的数据点数
ε = (max_score - min_score) / 50   # 防止价格停滞的小量
```

**迭代**（直到所有 job 被分配）：

**Step 1**：筛选活跃 worker

$$\text{active\_workers} = \{w \mid \text{count}[w] < \text{jobs\_per\_worker}\}$$

若 active_workers 为空，退出循环。

**Step 2**：每个活跃 worker 计算净收益

$$v_{w,j} = \text{score}[w][j] - \text{cost}[j]$$

扣掉当前价格后，这个数据点对该中心还值多少。

**Step 3**：每个活跃 worker 出价

每个 worker 找出净收益最高的前 `jobs_per_worker + 1` 个 job，对前 `jobs_per_worker` 个出价：

$$\text{bid}_{w,j} = v_{w,j_1} - v_{w,j_{p+1}} + \varepsilon \quad (p = \text{jobs\_per\_worker})$$

- $j_1$ 是该 worker 最想要的 job
- $j_{p+1}$ 是第 `p+1` 优选，作为"退而求其次"的参照基准
- 出价 = 最优与次优的差距 + ε，差距越大说明越想要，出价越高

**Step 4**：每个有人出价的 job 归属最高出价者

$$\text{new\_owner}[j] = \arg\max_w \text{bid}_{w,j}$$

- 若 `new_owner[j] ≠ assignment[j]`（归属发生变化）：
  - 原持有者 `count[原持有者] -= 1`，重新变为活跃
  - 新持有者 `count[新持有者] += 1`
  - `assignment[j] = new_owner[j]`

**Step 5**：更新价格

$$\text{cost}[j] += \max_w \text{bid}_{w,j}$$

**退出条件**：`assignment[j] ≥ 0` 对所有 j 成立（所有数据点都已分配）。

---

## 4. 为什么能保证均衡

三个机制共同作用：

**机制1：满额停止出价**

worker 持有 `jobs_per_worker` 个 job 后不再参与竞拍，不会无限扩张抢占其他 job。

**机制2：价格驱逐竞争**

热门 job 被反复竞争后价格 `cost[j]` 持续上涨，净收益 $v_{w,j} = \text{score}[w][j] - \text{cost}[j]$ 下降。当价格涨到某个临界点，其他 worker 发现"这个 job 已经不值得抢了"，转向次优选择。竞争压力自然分散。

**机制3：被驱逐者必须重新竞拍**

失去 job 的 worker 重新变为活跃，必须在下一轮继续出价。没有 worker 可以"躺平"，所有 worker 最终都会找到自己的 `jobs_per_worker` 个 job。

三个机制共同保证：**任何 worker 都不能长期持有超额 job，任何 job 都不会永久无人认领，算法最终收敛到全局均衡分配。**

## 5. 完整例子

### 设定

**6个数据点（job）**，**3个聚类中心（worker）**，每个中心分配 2 个点（`jobs_per_worker = 2`）。

距离矩阵：

```
        中心0   中心1   中心2
点0      1.0    3.0    8.0
点1      2.0    1.0    7.0
点2      3.0    2.0    1.0
点3      4.0    1.5    2.0
点4      9.0    2.0    1.5
点5      8.0    9.0    2.0
```

score = -distance，从 worker 视角（行=中心，列=数据点）：

```
        点0    点1    点2    点3    点4    点5
中心0  -1.0  -2.0  -3.0  -4.0  -9.0  -8.0
中心1  -3.0  -1.0  -2.0  -1.5  -2.0  -9.0
中心2  -8.0  -7.0  -1.0  -2.0  -1.5  -2.0
```

ε = (−1.0 − (−9.0)) / 50 = **0.16**

### 初始状态

```
cost       = [0,  0,  0,  0,  0,  0]
assignment = [-1, -1, -1, -1, -1, -1]
count      = [0,  0,  0]
```

---

### 第1轮

**Step 1**：三个中心都未满额，全部活跃。

**Step 2**：净收益 = score − cost（cost全0，净收益即score）。

**Step 3**：每个中心找 top 3（jobs_per_worker+1=3），出价给前2个。

```
中心0 top3: 点0(-1.0), 点1(-2.0), 点2(-3.0)
  出价 = (-1.0) - (-3.0) + 0.16 = 2.16，出价给 点0、点1

中心1 top3: 点1(-1.0), 点3(-1.5), 点2(-2.0)
  出价 = (-1.0) - (-2.0) + 0.16 = 1.16，出价给 点1、点3

中心2 top3: 点2(-1.0), 点4(-1.5), 点5(-2.0)
  出价 = (-1.0) - (-2.0) + 0.16 = 1.16，出价给 点2、点4
```

**Step 4**：各 job 的最高出价者：

```
点0: 中心0出价2.16          → 归属中心0
点1: 中心0出价2.16 > 中心1出价1.16  → 归属中心0
点2: 中心2出价1.16          → 归属中心2
点3: 中心1出价1.16          → 归属中心1
点4: 中心2出价1.16          → 归属中心2
点5: 无人出价               → 仍未分配
```

**Step 5**：更新价格：

```
cost = [2.16, 2.16, 1.16, 1.16, 1.16, 0]
```

**本轮结束状态**：

```
assignment = [0,  0,  2,  1,  2, -1]
count      = [2,  1,  2]   ← 中心0和中心2已满额
```

---

### 第2轮

**Step 1**：只有**中心1**未满额（count=1 < 2），活跃。

**Step 2**：中心1的净收益 = score[中心1] − cost：

```
点0: -3.0 - 2.16 = -5.16
点1: -1.0 - 2.16 = -3.16
点2: -2.0 - 1.16 = -3.16
点3: -1.5 - 1.16 = -2.66  ← 已持有
点4: -2.0 - 1.16 = -3.16
点5: -9.0 - 0    = -9.0
```

**Step 3**：中心1 top3：点3(-2.66)、点1(-3.16)、点2(-3.16)

```
出价 = (-2.66) - (-3.16) + 0.16 = 0.66，出价给 点3、点1
```

**Step 4**：
- 点3 已属于中心1，归属不变
- 点1 当前属于中心0，中心1出价0.66 > 0，**中心1抢走点1**
  - 中心0失去点1：count[0] = 1，重新变为活跃
  - 中心1得到点1：count[1] = 2，满额

**Step 5**：

```
cost[1] = 2.16 + 0.66 = 2.82
```

**本轮结束状态**：

```
assignment = [0,  1,  2,  1,  2, -1]
count      = [1,  2,  2]   ← 中心1和中心2满额，中心0重新活跃
```

---

### 第3轮

**Step 1**：只有**中心0**活跃（count=1 < 2）。

**Step 2**：中心0的净收益 = score[中心0] − cost：

```
点0: -1.0 - 2.16 = -3.16  ← 已持有
点1: -2.0 - 2.82 = -4.82
点2: -3.0 - 1.16 = -4.16
点3: -4.0 - 1.16 = -5.16
点4: -9.0 - 1.16 = -10.16
点5: -8.0 - 0    = -8.0
```

**Step 3**：中心0 top3：点0(-3.16)、点2(-4.16)、点1(-4.82)

```
出价 = (-3.16) - (-4.82) + 0.16 = 1.82，出价给 点0、点2
```

**Step 4**：
- 点0 已属于中心0，归属不变
- 点2 当前属于中心2，**中心0抢走点2**
  - 中心2失去点2：count[2] = 1，重新变为活跃
  - 中心0得到点2：count[0] = 2，满额

**Step 5**：

```
cost[0] = 2.16 + 1.82 = 3.98
cost[2] = 1.16 + 1.82 = 2.98
```

**本轮结束状态**：

```
assignment = [0,  1,  0,  1,  2, -1]
count      = [2,  2,  1]   ← 中心2重新活跃
```

---

### 第4轮

**Step 1**：只有**中心2**活跃（count=1 < 2）。

**Step 2**：中心2的净收益 = score[中心2] − cost：

```
点0: -8.0 - 3.98 = -11.98
点1: -7.0 - 2.82 = -9.82
点2: -1.0 - 2.98 = -3.98
点3: -2.0 - 1.16 = -3.16
点4: -1.5 - 1.16 = -2.66  ← 已持有
点5: -2.0 - 0    = -2.0
```

**Step 3**：中心2 top3：点5(-2.0)、点4(-2.66)、点3(-3.16)

```
出价 = (-2.0) - (-3.16) + 0.16 = 1.32，出价给 点5、点4
```

**Step 4**：
- 点5 未分配（assignment=-1），直接归属中心2：count[2] = 2，满额
- 点4 已属于中心2，归属不变

**Step 5**：

```
cost[5] = 0 + 1.32 = 1.32
```

**本轮结束状态**：

```
assignment = [0,  1,  0,  1,  2,  2]
count      = [2,  2,  2]   ← 全部满额，所有job已分配，退出
```

---

### 最终结果

```
中心0 → {点0, 点2}   总距离 = 1.0 + 3.0 = 4.0
中心1 → {点1, 点3}   总距离 = 1.0 + 1.5 = 2.5
中心2 → {点4, 点5}   总距离 = 1.5 + 2.0 = 3.5
全局总距离 = 10.0
```

每个中心恰好分配2个点，均衡约束满足。

---

### 竞争与驱逐的传导链

这个例子中发生了两次驱逐，体现了算法的核心动态：

```
中心1 抢走 点1（原属中心0）
  → 中心0 被迫转向，抢走 点2（原属中心2）
    → 中心2 被迫转向，抢走 点5（无人认领）
      → 所有job分配完毕
```

竞争压力像多米诺骨牌一样传导，最终让每个中心都找到了自己的位置。

## 6. 与普通 KMeans 的对比

| | 普通 KMeans | Auction Lap KMeans |
|---|---|---|
| 分配方式 | 每个点独立选最近中心 | 全局竞争，价高者得 |
| 均衡性 | 不保证，热门中心可能分配大量点 | 保证每个中心恰好 N/K 个点 |
| 码本利用率 | 可能出现空簇 | 每个码字都被使用 |
| 计算复杂度 | O(N·K) | O(N·K·迭代次数) |
| 适用场景 | 一般聚类 | 需要均衡分配的场景（如码本训练） |

普通 KMeans 的问题在于：如果数据分布不均匀，某些中心会吸引大量数据点，某些中心几乎没有数据点（码本坍塌）。Auction Lap 通过价格机制强制均衡，从根本上避免了这个问题。
