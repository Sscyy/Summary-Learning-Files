## 一、整体架构

```
hope_torch_distributed_launch.py
        │  解析集群信息 → 生成 torchrun 命令
        ▼
torchrun（PyTorch 内置启动器）
        │  为每个 GPU fork 出一个进程，注入环境变量
        ▼
run_mrl_distributed.py::main()   ← 每个 GPU 都运行这里
        │  读环境变量 → init_process_group → 加载模型 → 训练
        ▼
DistributedMRLTrainer
        │  DDP包装 → 训练循环 → AllGather → 损失 → 反向传播
```

## 二、启动层：hope_torch_distributed_launch.py

### 核心问题
在集群调度系统（AFO）上，怎么知道自己是第几台机器、master 在哪里？

### 两种模式

**单机模式**（AFO_SPEC 未注入）：
```python
node_rank   = 0
nnodes      = 1
master_addr = "127.0.0.1"
master_port = "29500"
```

**多机模式**（AFO_SPEC 存在）：
```python
afo_spec = json.loads(afo_spec_str)
workers   = afo_spec["cluster"]["worker"]   # ["ip1:port", "ip2:port", ...]
nnodes    = len(workers)                    # 总机器数
node_rank = int(afo_spec["taskId"])         # 我是第几台机器（0-based）
master_addr, master_port = workers[0].rsplit(":", 1)  # 第0台是 master
```

### 最终生成的命令
```bash
torchrun \
  --nproc_per_node=8 \      # 每台机器上的 GPU 数（nvidia-smi 探测）
  --nnodes=4 \              # 总机器数
  --node_rank=2 \           # 我是第2台机器
  --master_addr=10.0.0.1 \  # master 机器 IP
  --master_port=29500       # master 通信端口
```

> **注意**：这个脚本本身不启动训练，只生成 torchrun 命令字符串，由 shell 调用：
> `$(python hope_torch_distributed_launch.py) run_mrl_distributed.py`

## 三、torchrun 做了什么

`torchrun` 在每台机器上 fork N 个进程（N = GPU 数），并为每个进程注入环境变量：

```
机器0（node_rank=0）:        机器1（node_rank=1）:
  fork 进程0 → GPU0            fork 进程4 → GPU4
  fork 进程1 → GPU1            fork 进程5 → GPU5
  fork 进程2 → GPU2            fork 进程6 → GPU6
  fork 进程3 → GPU3            fork 进程7 → GPU7
```

每个进程被注入的环境变量：

| 环境变量 | 含义 | 例子（2机8卡，机器1的GPU2） |
|---------|------|--------------------------|
| `LOCAL_RANK` | 本机内的 GPU 编号 | `2` |
| `RANK` | 全局进程编号 | `6`（= 1×4 + 2） |
| `WORLD_SIZE` | 总进程数 | `8` |
| `MASTER_ADDR` | master 节点 IP | `10.0.0.1` |
| `MASTER_PORT` | master 端口 | `29500` |

## 四、进程初始化：run_mrl_distributed.py::main()

### 读取环境变量 + 绑定 GPU

```python
local_rank  = int(os.environ.get("LOCAL_RANK", 0))   # 本机 GPU 编号
global_rank = int(os.environ.get("RANK", 0))          # 全局进程编号
world_size  = int(os.environ.get("WORLD_SIZE", 1))    # 总进程数

torch.cuda.set_device(local_rank)
# 极其重要：让这个进程只"看到"自己的那块 GPU
# 如果不设置，所有进程默认都用 GPU0，显存爆炸
```

### init_process_group — 分布式通信的核心初始化

```python
dist.init_process_group(
    backend="nccl",
    timeout=timedelta(minutes=1200),
    # init_method 默认是 "env://"，读上面的环境变量
)
```

**backend 对比：**

| backend | 适用场景 | 底层技术 |
|---------|---------|---------|
| nccl | GPU训练 | NVLink/PCIe/InfiniBand |
| gloo | CPU训练 | TCP/IP |
| mpi | 超算集群 | MPI协议 |

**init_process_group 内部发生了什么：**
1. 每个进程向 master 节点（`MASTER_ADDR:MASTER_PORT`）注册自己的 RANK
2. master 收集所有进程的网络地址
3. 建立 world_size 个进程之间的通信拓扑
4. 初始化 NCCL communicator（GPU 间的通信句柄）

> **这一步是阻塞的**——所有进程必须都到达这里，才能继续。如果一台机器挂了，所有机器都会卡住然后超时。

## 五、数据层：DistributedSampler

```python
sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,  # 总 GPU 数
    rank=rank,                # 我是第几个 GPU
    shuffle=shuffle,
    seed=2333

```

### 数据分片机制

假设数据集 100 条，4 个 GPU：

```
完整数据集: [0, 1, 2, ..., 99]
                ↓ shuffle（所有 GPU 用同一个 seed，顺序一致）
打乱后:     [42, 7, 83, 15, 61, ...]
                ↓ 按 rank 取间隔索引
GPU0（rank=0）: 取位置 [0, 4, 8, 12, ...]
GPU1（rank=1）: 取位置 [1, 5, 9, 13, ...]
GPU2（rank=2）: 取位置 [2, 6, 10, 14, ...]
GPU3（rank=3）: 取位置 [3, 7, 11, 15, ...]
```

每个 GPU 看到的是**不重叠的子集**，合起来是完整数据集。

### 为什么必须调用 set_epoch？

```python
if hasattr(self.train_dataloader.sampler, 'set_epoch'):

```

`DistributedSampler` 内部用 `seed + epoch` 作为随机种子。如果不调用 `set_epoch`，每个 epoch 的数据顺序完全一样，等于没有 shuffle。

## 六、模型层：DDP 包装

```python
self.model = DDP(
    self.model,
    device_ids=[self.local_rank],      # 这个进程使用哪块 GPU
    output_device=self.local_rank,     # 输出放在哪块 GPU
)
```

### DDP 核心原理

```
前向传播（各自独立）：
  GPU0: batch0 → forward → loss0
  GPU1: batch1 → forward → loss1
  GPU2: batch2 → forward → loss2
  GPU3: batch3 → forward → loss3

反向传播（DDP 自动同步梯度）：
  各 GPU 算出各自的梯度 grad0, grad1, grad2, grad3
        ↓ DDP 自动触发 AllReduce（SUM ÷ world_size）
  所有 GPU: grad_avg = (grad0 + grad1 + grad2 + grad3) / 4
        ↓ 每个 GPU 用同一个梯度更新参数
  GPU0: params -= lr * grad_avg  ← 所有 GPU 参数保持一致！
  GPU1: params -= lr * grad_avg
  ...
```

**为什么 DDP 的 AllReduce 要除以 world_size？**

因为 DDP 的语义是"模拟大 batch 训练"：

```
单 GPU batch=32 的梯度：
  grad = (1/32) * Σ grad_i

DDP 4 GPU 各自 batch=8：
  GPU0: grad0 = (1/8) * Σ grad_i (i=0..7)
  ...
  AllReduce SUM: grad0+grad1+grad2+grad3 = 4 × (1/32) * Σ(全部32个)
  ÷ world_size(4): 得到 (1/32) * Σ(全部32个) ✓  等价于单卡 batch=32
```

### DDP 的 bucket 机制

DDP 不等所有层的梯度都算完才同步。当一个 bucket 满了（默认 25MB），立即触发这个 bucket 的 AllReduce，与后续层的反向传播**重叠执行**，隐藏通信延迟。

### get_unwrapped_model 为什么必要？

```python
def get_unwrapped_model(self, model):
    return model.module if hasattr(model, 'module') else model
```

DDP 包装后，`self.model` 是 `DDP` 对象，真正的模型在 `self.model.module`。保存模型、加载权重时必须用原始模型，否则会有 `module.` 前缀的 key 不匹配问题。

## 七、AllGather + AllReduce 完整推导

### 为什么需要 AllGather？

对比学习需要大量负样本。每个 GPU 只有 batch_size=8 的样本，负样本太少，模型学不好。

```
不用 AllGather（每 GPU 只有 8 个负样本）：
  GPU0: 8 个正样本 vs 7 个负样本 → 损失信号弱

用 AllGather（4 GPU × 8 = 32 个负样本）：
  GPU0: 8 个正样本 vs 31 个负样本 → 损失信号强 4 倍
```

### 场景设定

4 个 GPU，每个 GPU 的 batch_size = 2，embedding 维度 D：

```
GPU0 的本地 embedding: e0 = [a, b]
GPU1 的本地 embedding: e1 = [c, d]
GPU2 的本地 embedding: e2 = [e, f]
GPU3 的本地 embedding: e3 = [g, h]
```

---

### 第一步：AllGather 前向

```python
global_embeddings = all_gather_with_grad(local_embeddings)
```

每个 GPU 把自己的 [2, D] 广播出去，同时接收其他所有 GPU 的数据：

```
GPU0 执行后得到：E = [a, b, c, d, e, f, g, h]   shape [8, D]
GPU1 执行后得到：E = [a, b, c, d, e, f, g, h]   shape [8, D]
GPU2 执行后得到：E = [a, b, c, d, e, f, g, h]   shape [8, D]
GPU3 执行后得到：E = [a, b, c, d, e, f, g, h]   shape [8, D]
```

**4 个 GPU 现在持有完全相同的 E。**

```python
# AllGather.forward 核心代码
gathered_tensors = [torch.zeros_like(tensor) for _ in range(ctx.world_size)]
dist.all_gather(gathered_tensors, tensor)
gathered = torch.cat(gathered_tensors, dim=0)   # [B*world_size, D]
```

---

### 第二步：每个 GPU 独立计算损失

每个 GPU 用**本地的 query** 去和**全局的 key** 计算相似度矩阵：

```
GPU0（rank=0）：
  query = local_image_emb  shape [2, D]
  key   = global_text_emb  shape [8, D]
  sim   = query × key.T    shape [2, 8]
  labels = [0, 1]   # GPU0 第0张图对应全局第0段文字，第1张对应第1段
  L0 = cross_entropy(sim, labels=[0, 1])

GPU1（rank=1）：
  labels = [2, 3]   # GPU1 第0张图对应全局第2段文字
  L1 = cross_entropy(sim, labels=[2, 3])

GPU2: labels = [4, 5],  L2 = ...
GPU3: labels = [6, 7],  L3 = ...
```

**labels 为什么要加偏移？**

```python
labels = torch.arange(batch_size, device=self.device) + self.global_rank * batch_size
# GPU2, batch_size=2: labels = [4, 5]
# 含义：GPU2 的第0张图片，对应的正样本文字是全局第4个
```

全局 embedding 排列（all_gather 后）：
```
位置 0-1: GPU0 的 2 个样本
位置 2-3: GPU1 的 2 个样本
位置 4-5: GPU2 的 2 个样本  ← GPU2 的正样本在这里
位置 6-7: GPU3 的 2 个样本
```

**4 个 GPU 算的是不同的 loss，但每个 loss 都用了全局 8 个样本作为候选集。**

---

### 第三步：每个 GPU 独立执行 loss.backward()

每个 GPU 算出 `∂L_i/∂E`，shape `[8, D]`：

```
GPU0 算出 G0 = ∂L0/∂E，shape [8, D]
  G0[0] = ∂L0/∂a   # L0 关于向量 a 的梯度
  G0[1] = ∂L0/∂b
  G0[2] = ∂L0/∂c   # c 是 GPU1 的数据，但参与了 L0（作为负样本）
  G0[3] = ∂L0/∂d
  ...
  G0[7] = ∂L0/∂h

GPU1 算出 G1 = ∂L1/∂E，shape [8, D]
  G1[0] = ∂L1/∂a   # a 是 GPU0 的数据，但参与了 L1（作为负样本）
  G1[2] = ∂L1/∂c   # c 是 GPU1 自己的正样本
  ...
```

---

### 第四步：AllReduce SUM

```python
dist.all_reduce(grad_output, op=dist.ReduceOp.SUM)
```

每个 GPU 把自己的 Gi [8, D] 发出去，收回来的是所有人的 SUM：

```
G_total = G0 + G1 + G2 + G3   shape [8, D]

G_total[0] = ∂L0/∂a + ∂L1/∂a + ∂L2/∂a + ∂L3/∂a
           = ∂(L0+L1+L2+L3)/∂a
           = ∂L_global/∂a        ← a 在全局损失中的完整梯度 ✓

G_total[2] = ∂L_global/∂c        ← c 的完整梯度 ✓
...
```

**AllReduce 之后，每个 GPU 都持有完全相同的 G_total，且每一行都是对应向量在全局损失中的真实完整梯度。**

---

### 第五步：chunk 分割，每个 GPU 取自己那份

```python
grad_list = grad_output.chunk(ctx.world_size, dim=0)
# grad_list[0] = [∂L/∂a, ∂L/∂b]   shape [2, D]  ← 属于 GPU0
# grad_list[1] = [∂L/∂c, ∂L/∂d]   shape [2, D]  ← 属于 GPU1
# grad_list[2] = [∂L/∂e, ∂L/∂f]   shape [2, D]  ← 属于 GPU2
# grad_list[3] = [∂L/∂g, ∂L/∂h]   shape [2, D]  ← 属于 GPU3

grad_input = grad_list[ctx.rank]
```

**为什么不能取别人的梯度？**

```
GPU0 如果错误地用了 grad_list[2] = [∂L/∂e, ∂L/∂f]：

∂L/∂θ0 = ∂L/∂e × ∂e/∂θ0 + ∂L/∂f × ∂f/∂θ0

e 和 f 是 GPU2 的 embedding，由 θ2 产生，与 θ0 无关。
∂e/∂θ0 = 0，∂f/∂θ0 = 0  →  梯度全部归零，参数无法更新！
```

---

### 第六步：链式法则传回模型参数

```
GPU0 拿到 [∂L/∂a, ∂L/∂b] 之后，PyTorch 继续自动微分：

∂L/∂θ0 = ∂L/∂a × ∂a/∂θ0 + ∂L/∂b × ∂b/∂θ0
              ↑                    ↑
     AllGather backward 算出来    前向传播建立的计算图自动算
```

---

### 完整流程图

```
前向：
  GPU0: [a,b] ──┐
  GPU1: [c,d] ──┤ AllGather ──→ 每个GPU都有 E=[a,b,c,d,e,f,g,h]
  GPU2: [e,f] ──┤                         ↓
  GPU3: [g,h] ──┘              各自算 L0, L1, L2, L3（用全局E）

反向：
  GPU0: loss.backward() → G0=[∂L0/∂a,...,∂L0/∂h]  shape [8,D]
  GPU1: loss.backward() → G1=[∂L1/∂a,...,∂L1/∂h]  shape [8,D]
  GPU2: loss.backward() → G2=[...]
  GPU3: loss.backward() → G3=[...]
            ↓
       AllReduce SUM
            ↓
  每个GPU都有 G_total=[∂L/∂a, ∂L/∂b, ..., ∂L/∂h]  shape [8,D]
            ↓
       chunk(4, dim=0)
            ↓
  GPU0 取 [∂L/∂a, ∂L/∂b] → 链式法则 → ∂L/∂θ0 → 更新参数
  GPU1 取 [∂L/∂c, ∂L/∂d] → 链式法则 → ∂L/∂θ1 → 更新参数
  GPU2 取 [∂L/∂e, ∂L/∂f] → 链式法则 → ∂L/∂θ2 → 更新参数
  GPU3 取 [∂L/∂g, ∂L/∂h] → 链式法则 → ∂L/∂θ3 → 更新参数
```

---

### AllGather backward 的 AllReduce 为什么不除 world_size？

**DDP 的 AllReduce** 定义的损失是：
```
L_DDP = (L0 + L1 + L2 + L3) / 4    ← 归一化，模拟大 batch
→ 梯度需要除以 world_size
```

**AllGather 对比学习的损失**：
```
L_global = L0 + L1 + L2 + L3       ← 直接求和，不归一化

∂L_global/∂a = ∂L0/∂a + ∂L1/∂a + ∂L2/∂a + ∂L3/∂a

这是 a 在全局损失中的完整梯度，AllReduce SUM 的结果就是正确答案，
不需要再除 4。
```

**关于梯度偏大的问题：**

```
cross_entropy 内部对 batch 维度求平均：
  L0 = (1/2) * [CE(sim[0], label=0) + CE(sim[1], label=1)]

L_global = L0+L1+L2+L3
         = (1/2) * Σ(全部8个CE)
         = 4 × (1/8) * Σ(全部8个CE)
         = 4 × 单卡 batch=8 的损失
```

L_global 是单卡损失的 4 倍，梯度也是 4 倍，等效于把学习率放大了 4 倍。这在多卡对比学习（MoCo、CLIP）中是标准做法，实际使用时学习率需要相应调整。

---

### DDP AllReduce vs AllGather backward AllReduce 对比

| | DDP 的 AllReduce | AllGather backward 的 AllReduce |
|--|--|--|
| 同步的是什么 | 模型参数的梯度 `∂L/∂θ` | embedding 的梯度 `∂L/∂e` |
| 为什么要 SUM | 各 GPU 算的是同一参数梯度的不同部分 | 各 GPU 的 loss 都依赖同一个 embedding |
| 要不要除 world_size | 要（等价大batch语义） | 不要（是真实完整梯度之和） |
| 发生时机 | `loss.backward()` 时 DDP 自动触发 | 自定义 `AllGather.backward` 里手动触发 |

## 八、InfoNCE Loss 与 cross_entropy 的等价性

### InfoNCE Loss 公式

$$\mathcal{L}_{InfoNCE} = -\frac{1}{N}\sum_{i=1}^{N} \log \frac{\exp(q_i \cdot k_i^+ / \tau)}{\sum_{j=1}^{N} \exp(q_i \cdot k_j / \tau)}$$

- $q_i$：第 $i$ 个 query（图片 embedding）
- $k_i^+$：第 $i$ 个正样本 key（对应文字 embedding）
- $k_j$：所有 N 个 key（包括正样本和负样本）
- $\tau$：temperature

### 代码实现

```python
similarity_i2t = torch.matmul(img_emb_d, global_txt_emb_d.T) / self.temperature
# shape: [local_batch, global_batch]

loss_i2t = F.cross_entropy(similarity_i2t, labels)
```

### 展开计算过程

以 GPU0（rank=0，batch=2，全局8个样本）为例：

```
similarity_i2t shape [2, 8]：
       t0      t1      t2      t3      t4      t5      t6      t7
  a  [a·t0/τ, a·t1/τ, a·t2/τ, a·t3/τ, a·t4/τ, a·t5/τ, a·t6/τ, a·t7/τ]
  b  [b·t0/τ, b·t1/τ, b·t2/τ, b·t3/τ, b·t4/τ, b·t5/τ, b·t6/τ, b·t7/τ]

labels = [0, 1]
```

`F.cross_entropy(logits, labels)` 展开：

```
对图片 a（label=0）：
  loss_a = -log( exp(a·t0/τ) / Σ_j exp(a·tj/τ) )
         = -(a·t0/τ) + log(Σ_j exp(a·tj/τ))

对图片 b（label=1）：
  loss_b = -(b·t1/τ) + log(Σ_j exp(b·tj/τ))

L0 = (loss_a + loss_b) / 2
```

### 与 InfoNCE 完全对应

```
InfoNCE 第 i 项：
  -log( exp(q_i·k_i+/τ) / Σ_j exp(q_i·kj/τ) )

cross_entropy 第 i 行：
  -log( exp(sim[i, label_i]) / Σ_j exp(sim[i, j]) )

其中 sim[i, label_i] = q_i · k_{label_i} / τ = q_i · k_i+ / τ
```

**`F.cross_entropy` 就是 InfoNCE loss 的向量化实现，labels 数组指定了每个 query 对应的正样本位置。**

### temperature τ 的作用

```
τ 大（如 0.5）：logits 平滑，softmax 分布均匀，梯度小，学习慢
τ 小（如 0.07）：logits 尖锐，正样本必须明显高于负样本，梯度大，学习快但不稳定

代码里 τ=0.07 是 CLIP 论文的标准设置。
```

## 九、MRL（Matryoshka Representation Learning）

### 核心思想

同一个模型输出的 embedding 的**前 N 维**，本身就是一个高质量的 N 维 embedding。这样一个模型可以支持不同精度需求的检索场景。

### 代码实现

```python
for dim in self.nesting_dims:  # 比如 [128, 256, 512, 1536]
    # 截取前 dim 维（嵌套结构）
    img_emb_d = image_embeddings[:, :dim]             # [B, dim]
    global_img_emb_d = global_image_embeddings[:, :dim]  # [B*world_size, dim]

    # 归一化
    img_emb_d = F.normalize(img_emb_d, p=2, dim=1)
    global_img_emb_d = F.normalize(global_img_emb_d, p=2, dim=1)

    # 相似度矩阵
    similarity_i2t = torch.matmul(img_emb_d, global_txt_emb_d.T) / self.temperature
    # [B, B*world_size]

    loss_d = (F.cross_entropy(similarity_i2t, labels) +
              F.cross_entropy(similarity_t2i, labels)) / 2.0

    total_loss += loss_d

total_loss = total_loss / len(self.nesting_dims)  # 对所有维度取平均
```

## 十、GradCache — 显存优化技术

### 问题背景

对比学习需要大 batch（负样本越多越好），但大 batch 显存不够。GradCache 用"分步计算"解决这个矛盾。

### 三阶段执行流程

**阶段一：无梯度前向（`_forward_no_grad`）**

```python
for batch in batches:   # gc_num_steps 个 mini-batch
    with torch.no_grad():
        pooled = model(batch)   # [B, D]

    pooled = pooled.detach().requires_grad_(True)
    # detach：断开与模型参数的计算图连接
    # requires_grad_(True)：让 pooled 成为叶子节点，可以接收梯度
    # 效果：pooled 可以接收来自损失的梯度，但梯度不会流向模型参数
```

**阶段二：在大 embedding 上计算损失和梯度**

```python
image_embeddings = torch.cat(image_emb_list, dim=0)  # [B*gc_n, D]
loss = mrl_contrastive_loss(image_embeddings, text_embeddings)
loss.backward()
# 此时 image_embeddings.grad 有值了（shape [B*gc_n, D]）
# 但模型参数的 grad 仍为 None（因为 detach 了）

# 把大 batch 的梯度切回 mini-batch 大小
image_grad_list = image_embeddings.grad.split(mini_batch_size, dim=0)
```

**阶段三：带梯度的重新前向（`_forward_with_grad`）**

```python
for img_inputs, img_grad in zip(image_inputs_list, image_grad_list):
    pooled = model(img_inputs)   # 重新前向，有梯度
    pooled.backward(img_grad)    # 用缓存的梯度作为种子

# 链式法则：
# d(loss)/d(params) = d(loss)/d(pooled) × d(pooled)/d(params)
#                   =    img_grad       ×  这次前向建立的计算图
```

### 显存对比

```
普通训练（batch=32）：
  模型激活值：32 × seq_len × hidden_size × 层数 → 很大

GradCache（gc_num_steps=4, mini_batch=8）：
  阶段一：每次只保留 8 个样本的 embedding（无梯度，极小）
  阶段二：只有 embedding 矩阵 [32, D]（很小）
  阶段三：每次只有 8 个样本的激活值
  → 显存峰值 ≈ 普通训练的 1/4
```

## 十一、同步屏障：dist.barrier()

`barrier()` 是全局同步点，所有进程都必须到达这里，才能继续。

| 位置 | 原因 |
|------|------|
| `train_step` 中 embedding 提取后 | 确保所有 GPU 都完成了 embedding 提取，再进行 all_gather |
| `train_epoch` 开始前 | 确保所有 GPU 都完成了上一步的初始化 |
| 保存检查点后 | 只有 rank=0 在写文件，其他 GPU 等待，防止文件写入冲突 |
| 评估后 | 等所有 GPU 都评估完再继续训练 |

## 十二、混合精度训练

```python
self.autocast_dtype = torch.bfloat16 if config.get('bf16', True) else torch.float16

with torch.amp.autocast(device_type='cuda', dtype=self.autocast_dtype):
    outputs = self.model(**inputs, output_hidden_states=True, return_dict=True)
```

- **bfloat16**：数值范围和 float32 相同，精度略低，对训练稳定性更友好
- **float16**：精度更高，但数值范围小，容易溢出，需要配合 GradScaler
