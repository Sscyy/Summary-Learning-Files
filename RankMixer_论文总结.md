# RankMixer 论文总结

> **一句话概括**：通过硬件感知的架构设计，在保持推理延迟不变的前提下，将推荐排序模型参数扩大 70-100 倍。

---

## 一、背景：为什么要做这件事？

### 1.1 工业推荐的独特约束
| 约束         |       说明 |
|-------------|----------|
| **延迟敏感** | 在线服务必须满足严格的 P99 延迟（如 <20ms） |
| **高 QPS** | 每秒处理百万级请求，计算效率至关重要 |
| **特征异构** | 数百个特征字段，包含 ID、统计、序列等完全不同的语义空间 |

### 1.2 现有方法的瓶颈
```
传统方法（DCN、DeepFM、DHEN 等）：
  - 设计于 CPU 时代，使用手工设计的特征交叉算子
  - 算子是"内存带宽受限"（memory-bound）而非"计算受限"
  - GPU 利用率极低：MFU 仅 4-5%（理论峰值 50-80%）
  
简单堆叠参数：
  - 收益有限甚至为负
  - 参数增长 → FLOPs 增长 → 延迟增长（线性关系）
```

### 1.3 核心矛盾
> **Scaling Law 告诉我们"越大越好"，但工业场景要求"越快越好"**

## 二、核心思想：硬件优先的架构设计

### 2.1 设计哲学转变
| 传统方法 | RankMixer |
|---------|-----------|
| 先设计表达力强的模型，再优化推理 | 以 GPU 效率为第一性原理指导架构设计 |
| 追求数学上的特征交叉完备性 | 追求计算上的高并行、高 MFU |
| 参数增长 ∝ 计算增长 ∝ 延迟增长 | **参数 ∝ 计算 ∝ 延迟 三重解耦** |

### 2.2 什么是高 MFU？
```
MFU (Model FLOPs Utilization) = 实际达到的 FLOPs / GPU 理论峰值 FLOPs

低 MFU 的原因：
  - 小矩阵乘法（shape 太小，GPU 并行度不足）
  - 不规则内存访问（attention 的稀疏访问模式）
  - 算子碎片化（多种不同类型算子，无法融合）

RankMixer 目标：所有核心操作都是大矩阵乘法（GEMM）
```

## 三、方法详解：数据如何流转

### 3.1 整体架构（宏观视角）
```
输入特征 (300+ 个) 
    ↓
Tokenization ──► T 个固定维度的特征 tokens
    ↓
重复 L 层：
  ├─ Multi-head Token Mixing（跨 token 信息融合，零参数）
  └─ Per-token FFN（独立子空间建模，参数不共享）
    ↓
Mean Pooling ──► 输出表示 ──► CTR/CVR 预测
```

### 3.2 第一层：Tokenization（语义对齐）
```
为什么：数百个异构特征维度不一，需要统一表示

怎么做：
  1. 按语义分组（用户画像/视频信息/统计特征/序列特征）
  2. 组内 Concat 成统一长向量
  3. 等长切分（如 16 段）
  4. 投影到 D 维（所有 token 共享同一个投影层）

关键收益：
  - 维度对齐：所有 token 同维度，可批量化计算
  - GPU 高效：T×D 的矩阵运算充分利用并行度
  - 语义保留：同组特征相邻，跨组信息也有机会混合
```

### 3.3 第二层：Multi-head Token Mixing（信息融合）
```
为什么：需要跨特征交互，但传统方法有问题
  - Self-Attention：内积相似度对异构特征无意义，且 O(T²) 复杂
  - 手工交叉算子（DCN/FM）：GPU 效率低

怎么做（核心创新）：
  1. 每个 token 切分为 H 个 heads（H=T）
  2. 重排：第 h 个新 token = 所有旧 token 的第 h 个 head
  3. 残差连接 + LayerNorm

关键特性：
  - 零参数：纯数据重排，无学习参数
  - O(T·D) 复杂度：远低于 Attention 的 O(T²·D)
  - 完美向量化：reshape + concat 是 GPU 友好操作

收益：实现跨特征交互的同时，MFU 接近理论峰值
```

### 3.4 第三层：Per-token FFN（独立建模）
```
为什么：异构特征混合建模会导致"高频特征主导"

怎么做：
  每个 token 拥有独立的 FFN 参数（不共享）
  v_t = FFN_t(s_t) = W_{t,2} · GELU(W_{t,1} · s_t)

关键洞察：
  - 参数隔离：用户画像、视频信息、统计特征各自走自己的网络
  - 参数扩展：T 个独立 FFN 使参数量×T，但 FLOPs 不变
    （因为每个 token 只走自己的 FFN，不增加计算量）

避免表示坍缩：
  - Token Mixing 后，各 token 已包含混合信息
  - 如果共享 FFN，容易映射到相似表示
  - 独立 FFN 保持各 token 的语义区分度
```

### 3.5 第四层：Sparse MoE 扩展（可选）
```
目标：进一步扩展到 1B+ 参数

挑战：传统 MoE 在 RankMixer 中失效
  1. Uniform k-expert：所有 token 强制选 k 个专家，浪费计算
  2. Expert under-training：T×N_e 个专家，梯度分散

解决方案：DTSI-MoE（Dense-training, Sparse-inference）
  ├─ 训练时：Dense（所有专家参与，Softmax 加权）
  │   目的：确保所有专家获得梯度，避免死亡专家
  │
  └─ 推理时：Sparse（ReLU 路由，仅激活部分专家）
      特点：动态专家数（高信息 token 激活更多）
      收益：参数扩展 8 倍，计算量仅增加 1 倍

负载均衡：L_reg = Σ G_{i,j}，控制平均激活比例
```

## 四、关键创新点（技术贡献）

| 创新 | 解决的问题 | 核心价值 |
|------|-----------|---------|
| **Multi-head Token Mixing** | 异构特征的高效交互 | 零参数、O(TD) 复杂度、GPU 友好 |
| **Per-token FFN** | 异构特征空间隔离 | 避免高频主导、参数与计算解耦 |
| **DTSI-MoE** | 专家极化与训练不足 | 训练密集保证充分学习，推理稀疏保证高效 |
| **ReLU Routing** | 固定 Top-K 的资源浪费 | 动态专家分配，高信息 token 用更多计算 |

## 五、实验结果

### 5.1 离线性能（Douyin 数据集）
| 模型 | Finish AUC | Skip AUC | 参数量 | FLOPs |
|------|------------|----------|--------|-------|
| DLRM-MLP | 0.8554 | 0.8124 | 8.7M | 52G |
| DHEN | +0.18% | +0.36% | 22M | 158G |
| Wukong | +0.29% | +0.49% | 122M | 442G |
| **RankMixer-100M** | **+0.64%** | **+0.86%** | 107M | 233G |
| **RankMixer-1B** | **+0.95%** | **+1.25%** | 1B | 2.1T |

### 5.2 在线部署效果（核心指标）
| 指标 | 基线 | RankMixer-1B | 变化 |
|------|------|--------------|------|
| 密集参数量 | 15.8M | 1.1B | **↑ 70×** |
| MFU | 4.47% | 44.57% | **↑ 10×** |
| 推理延迟 | 14.5ms | 14.3ms | **↓ 1.4%** |
| 用户活跃天数 | - | - | **+0.2%** |
| App 使用时长 | - | - | **+0.5%** |

### 5.3 Scaling Law
- RankMixer 的参数扩展曲线和 FLOPs 扩展曲线都是最陡峭的
- 相同 FLOPs 下，AUC 增益最高
- 相同 AUC 增益下，所需 FLOPs 最少

## 六、优缺点与局限性

### 6.1 优势
| 优势 | 说明 |
|------|------|
| **硬件对齐** | 所有算子都是 GPU 友好的大矩阵乘法，MFU 高达 45% |
| **参数-计算-延迟三重解耦** | 参数扩大 70 倍，延迟反而下降 |
| **工业落地验证** | Douyin 全流量部署，5 个月在线实验 |
| **通用性强** | 推荐和广告场景均验证有效 |

### 6.2 局限性与待验证问题
| 局限 | 说明 |
|------|------|
| **领域依赖** | 架构针对推荐特征设计，迁移到 NLP/CV 需验证 |
| **工程依赖** | 低延迟依赖 Op fusion、混合精度等工程优化 |
| **Tokenization 设计** | 语义分组需要领域知识，自动化方法未探讨 |
| **专家极化** | DTSI 缓解但未完全解决，长期训练稳定性待观察 |
| **长序列** | 未处理超长用户行为序列（>1000） |

## 八、核心启示

1. **Scaling Law 的成功需要硬件对齐**：不是"模型越大越好"，而是"模型越适合硬件越好"
2. **参数与计算可以解耦**：通过架构设计，增加参数不一定增加 FLOPs
3. **异构性需要显式建模**：推荐系统的异构特征需要参数隔离，而非统一处理
4. **训练与推理可以不对称**：DTSI 策略实现了"训练充分"与"推理高效"的平衡
